Spec-driven planning for AI-assisted development

A lightweight intent layer that keeps your project visible to you, your team, and your AI assistant. Markdown and JSON, no PM tool required.

Problem

You ship fast. You lose track faster.

A few weeks of AI-assisted velocity and the project becomes opaque to its own author.

  • Dozens of *_PLAN.md files at the repo root. Active, stale, shipped, all mixed.
  • No single answer to "what's in flight right now?"
  • Plans that quietly contradict each other.
  • Every new session starts with 15 minutes of re-explaining.
repo root

FEATURE_A_PLAN.md
FEATURE_B_PLAN.md (stale?)
FEATURE_C_PLAN.md (shipped?)
OLD_IDEA_PLAN.md (abandoned?)
NEW_PLAN.md (conflicts?)

Nobody knows.

Insight

The problem isn't the AI. It's that intent has no structure.

Without structure, no one can reason about where a project stands. Not you, not your team, and not your AI assistant.

Plans

Intent layer. What you're building, why, and what phase it's in.

Git log

Ground truth. What actually shipped, and when.

/plans-sync

Reconciles plans against git log on a schedule. Catches drift.

Primary audience: you. Your AI assistant benefits as a side effect of structured input.

Scope

What this is, and what it isn't.

What it is
  • An intent layer for projects with multiple features in flight.
  • Plain markdown and JSON in your repo, git-excluded by default.
  • Persistent context across plans, for yourself, over time.
  • Reading order: STATUS.md first, then plan files.
What it isn't
  • Not a PM tool. (Linear, Jira, Asana stay where they are.)
  • Not a task tracker. Operates at plan and phase level.
  • Not a team collaboration system.
  • Not a replacement for design docs, RFCs, or ADRs.
  • Not git-tracked or auditable.
Audience

Is this for you?

Adopt when

  • Solo or small team (1 to 3 people).
  • 3+ committed features in parallel.
  • You're losing track of in-flight vs shipped.
  • Plans are accumulating at the repo root.
  • You re-explain context every AI session.

Skip when

  • Single-feature scope.
  • Team of 5+. Use a real PM tool.
  • Regulated environments. You need an audit trail.
  • Open-source projects. Use Issues.
  • One-off scripts and prototypes.
Structure

One directory. Five files. Two folders.

plans/
  README.md           Onboarding for anyone new to the repo
  STATUS.md           Daily check-in: in flight, up next, shipped, backlog
  plans.json          Machine-readable snapshot, auto-generated
  roadmap.html        Interactive dashboard, reads plans.json directly
  active/             In-progress and committed plans
  shipped/            Completed plans, kept for history

Git-excluded

Lives in your repo, never committed. Local to your machine.

echo "plans/" >> .git/info/exclude

Plain files

Markdown and JSON only. No database, no account, no external service to depend on.

Anatomy

Every plan has two layers.

Frontmatter is for tooling. The status banner is for humans. They must agree.

FrontmatterYAML, 7 required fields
---
status: active
priority: P1
owner: you
type: feature
depends_on: [AUTH_PLAN]
blocks: [DASHBOARD_PLAN]
last_updated: 2026-05-04
---
Status bannermarkdown, second heading
## Status

- **Overall:** In progress, Phase 2 of 3
- **Phase 0 (Design):** Done (2026-04-28)
- **Phase 1 (Backend):** Done (2026-05-01)
- **Phase 2 (Frontend):** In progress
- **Phase 3 (Tests):** Not started

**Next action:** Wire up the API client.
**Last updated:** 2026-05-04
Two-source rule: if frontmatter and banner disagree, frontmatter wins. Update both together.
Lifecycle

One capture point. One home per stage.

The rule that prevents scattered docs: every idea starts as a backlog bullet, and only graduates when work is committed.

1
New idea
Bullet in STATUS.md backlog. No plan file yet, work isn't committed.
2
Committed
Create plans/active/FEATURE.md. Add a row to STATUS.md "Up next".
3
Work starts
Set in_flight: true. Move STATUS.md row to "In flight".
4
Phase ships
Mark phase Done in the banner. Update STATUS.md. Bump last_updated.
5
Fully done
mv plans/active/FEATURE.md plans/shipped/. Move STATUS.md row to "Recently shipped".

Reconciliation

Plans drift from reality. That's why /plans-sync exists.

Plans describe intent. Git log records reality. Without reconciliation, your planning system becomes misleading rather than helpful.

  • Shipped code, plan still "Not started".
  • Active plan, no commits in 14 days.
  • STATUS.md row with no underlying file.
  • Frontmatter and banner disagree.
  • Dependency edges that don't reciprocate.
Coming

/plans-sync

  • Reads all plan frontmatter.
  • Runs git log since last sync.
  • Detects drift automatically.
  • Regenerates plans.json and STATUS.md.
  • Shows a diff. Asks before writing.

15-minute manual audit becomes 2-minute review-and-confirm.

AI as side effect

The system is for you. The AI just benefits from structure.

Without this system

  • Re-explain context every session.
  • AI suggests things you decided against last week.
  • Work gets duplicated.
  • 15 minutes of context-setting before real work.

With this system

  • AI reads STATUS.md at session start.
  • "What's next?" has a deterministic answer.
  • AI knows which plan to update when you ship.
  • Structured input means better reasoning.

The integration is two lines in CLAUDE.md (or AGENTS.md, .cursorrules): point to STATUS.md, describe the lifecycle.

Quick start

Two commands. The rest is your first plan.

curl -sSL https://raw.githubusercontent.com/yrangana/Plans/main/install.sh | bash
plans-init /path/to/your/project
1
Install
Clones to ~/.local/share/plans and symlinks plans-init and plans-update to ~/.local/bin/. Re-run any time to upgrade.
2
Bootstrap your project
plans-init copies the template into plans/, adds it to .git/info/exclude, and offers to append the planning section to your AI instruction file (CLAUDE.md, AGENTS.md, .cursorrules, .windsurfrules).
3
Replace the example plan
Edit plans/active/EXAMPLE_PLAN.md with your real first plan. Add a row to STATUS.md.
4
Open the dashboard
python -m http.server 8080, then visit http://localhost:8080/plans/roadmap.html.
5
Stay current
plans-update pulls dashboard improvements without touching your data. Backups go to .bak.
