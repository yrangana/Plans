The Problem

You're building fast with AI. A few weeks in:

  • 😩 Dozens of *_PLAN.md files at the root. Active? Stale? Shipped?
  • ❓ No single place to answer "what's in flight?"
  • ⚡ Plans that contradict each other
  • 🔄 Re-explaining context at the start of every session
  • ⏱️ 15 minutes of context-setting before real work

Typical repo root

📄 FEATURE_A_PLAN.md

📄 FEATURE_B_PLAN.md (stale?)

📄 FEATURE_C_PLAN.md (shipped?)

📄 OLD_IDEA_PLAN.md (abandoned?)

📄 NEW_PLAN.md (conflicts?)

Nobody knows.

The Insight

The problem isn't the AI. The problem is that project intent has no structure.
Without structure, nothing (human or AI) can reason about where things stand.

📋 Plans

Intent layer: what we're building, why, and what phase it's in

🔀 Git log

Ground truth: what actually shipped and when

⚡ /status-sync

Reconciles the two weekly, catches drift automatically

Primary audience: you. You open STATUS.md and know where you stand. Your AI assistant benefits too. Structured files beat scattered docs every time.

What This Is, What It Isn't

✅ What this is

  • Intent layer for projects with multiple features in flight
  • Plain markdown + JSON in your repo (git-excluded)
  • Persistent context across plans, for yourself, over time
  • Reading order: STATUS.md first, then plan files

❌ What it is not

  • Not a PM tool (not Linear, Jira, Asana)
  • Not a task tracker (no individual TODOs)
  • Not a team collaboration system
  • Not a replacement for design docs / RFCs
  • Not git-tracked or auditable

🎁 What you get

  • Single source of truth for "what's in flight"
  • Persistent context for AI sessions
  • Shareable visual roadmap for stakeholders
  • Drift detection between plans and git log
  • Forced clarity, surfaces muddy thinking early
  • Lower cost when resuming after a break

Is This For You?

👤 Adopt when

  • Solo dev or small team (1 to 3 people)
  • 3+ features committed in parallel
  • AI-assisted development
  • Plans accumulating at repo root
  • You're losing track of what's in flight vs shipped

🚫 Skip when

  • Single-feature scope
  • Team of 5+ (use real PM tool)
  • Regulated environments (need audit trail)
  • Open source projects (use Issues)
  • One-off scripts and prototypes

The Structure

plans/
├── STATUS.md          ← Your daily check-in. Open this, not a PM tool.
├── plans.json         ← Machine-readable snapshot (auto-generated)
├── roadmap.html       ← Shareable visual dashboard
├── active/            ← In-progress and committed plans
│   └── MY_FEATURE.md
└── shipped/           ← Completed plans (kept for history)

Git-excluded

Lives in the repo, never committed. Local to your machine.

echo "plans/" >> .git/info/exclude

Plain files

Markdown + JSON. No database, no account, no external tool.

Plan File Anatomy

Two layers, both required, both must agree.

Frontmatter (for tooling)

---
status: active
priority: P1
owner: yash
type: feature
depends_on: [AUTH_PLAN]
blocks: [DASHBOARD_PLAN]
last_updated: 2026-05-04
---

Status banner (for humans)

## Status

- **Overall:** 🟡 Phase 2 of 3
- **Phase 0 (Design):** ✅ Done
- **Phase 1 (Backend):** ✅ Done
- **Phase 2 (Frontend):** 🟡 Active
- **Phase 3 (Tests):** 🔴 Not started

**Next action:** Wire up API client.
**Last updated:** 2026-05-04

⚖️ Two-source rule: frontmatter and banner must agree. If they conflict, frontmatter wins.

Idea Lifecycle: One Capture Point

The rule that prevents scattered docs: everything starts as a backlog bullet.

💡
New idea
Bullet in STATUS.md backlog. Nothing else, not committed yet.
📄
Committed
Create plans/active/FEATURE.md + STATUS.md "Up next" row
🟡
Work starts
Set in_flight: true → move to STATUS.md "In flight"
✅
Phase ships
Mark ✅ in banner → update STATUS.md → bump last_updated
📦
Fully done
mv plans/active/ plans/shipped/ → STATUS.md "Recently shipped"

Git Is the Ground Truth

Plans describe intent. Git log is what actually happened. They drift.

  • ⚠️ Shipped code, plan still 🔴
  • ⚠️ Active plan, no commits in 14 days
  • ⚠️ Plan in STATUS.md, no file exists
  • ⚠️ Frontmatter vs banner disagree
  • ⚠️ Orphaned dependency edges
Left undetected, drift makes your planning system misleading rather than helpful.

Coming: /status-sync skill

  • 📖 Reads all plan frontmatter
  • 🔀 Runs git log since last sync
  • 🔍 Detects drift automatically
  • 🔄 Regenerates plans.json + STATUS.md
  • 📋 Shows diff, asks before writing

15-min manual weekly audit → 2-min review-and-confirm

Without this system

With this system
